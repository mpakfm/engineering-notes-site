# Рабочие заметки инженера

Личный сайт Сергея Фомина: светлый инженерный блокнот для заметок, разборов и чек-листов о веб-разработке, Bitrix, интеграциях, релизах, оценке задач и процессах в команде.

## Суть проекта

Это не портфолио и не маркетинговый блог. Сайт должен восприниматься как публичная инженерная тетрадь: спокойно, практично, структурно, без пафоса.

Главная идея:

> Не идеально, но понятно и работает.

## Комментарии по мобильному референсу для верстки

Мобильный макет считать основным референсом для адаптива. Он сохраняет характер desktop-версии и выглядит как полноценная mobile-first версия инженерного блокнота.

## Что важно сохранить на mobile

- Одноколоночную структуру.
- Компактную шапку с логотипом `mp/`, wordmark `mpakfm`, подписью и burger menu.
- Hero-блок в стиле блокнота: теплый фон, тонкая сетка, заголовок и ключевая фраза.
- Порядок блоков:
  1. Header
  2. Hero
  3. С чего начать
  4. Свежие заметки
  5. Избранные чек-листы
  6. Темы
  7. Коротко обо мне
  8. Footer
- Теплую светлую палитру, muted green акцент и мягкие карточки.

## Что адаптировать, а не копировать слепо

Hero нужно проверять на ширинах 360–390 px: при тесноте упрощать декоративные элементы. В карточках заметок метаданные должны идти под описанием (теги, дата, время чтения), а не правой колонкой.

## UX-требования для mobile

- Не показывать desktop-меню целиком, использовать burger menu.
- Не добавлять неработающие элементы. Иконку поиска из референса не выводить, пока поиск не реализован.
- Поддерживать читаемость: основной текст около 15–16 px.
- Не перегружать декоративной «тетрадностью».
- Карточки должны выглядеть нажимаемыми.
- Между секциями нужен воздух.

## Работа через Web-Codex со смартфона

Один из смыслов проекта — проверить рабочий сценарий, в котором идею можно сформулировать с телефона, отдать Web-Codex, получить изменения в GitHub и после merge увидеть результат на сайте.

Web-Codex должен работать не на production-сервере, а в репозитории GitHub:

1. Взять свежий `master`.
2. Создать отдельную ветку под задачу.
3. Внести изменения в проект.
4. Запустить проверку сборки.
5. Сделать commit.
6. Push ветки в GitHub.
7. Открыть pull request в `master`.

После merge в `master` деплой выполняется через CI/CD. На сервер руками ходить не нужно.

## Как добавить новую заметку

Новая заметка добавляется как Markdown-файл в `src/content/notes/`.

Имя файла должно быть латиницей в `kebab-case`, потому что из имени файла получается URL:

```text
src/content/notes/zachem-ya-zavel-inzhenernyy-bloknot.md
```

Такой файл будет доступен по адресу:

```text
/notes/zachem-ya-zavel-inzhenernyy-bloknot/
```

В начале файла нужен frontmatter:

```md
---
title: "Название заметки"
description: "Короткое описание для карточки и списка заметок."
pubDate: 2026-05-17
tags: ["Bitrix", "Git", "AI-агенты"]
readingTime: "5 мин"
draft: false
---
```

После frontmatter идет текст заметки в Markdown.

Если `draft: false`, заметка автоматически попадает:

- на страницу `/notes/`;
- на детальную страницу `/notes/<slug>/`;
- на главную в блоки с заметками по текущей логике.

Если `draft: true`, заметка не выводится публично и не участвует в списках.

## Как выбирать темы и теги

Темы сейчас берутся из поля `tags` опубликованных заметок. Отдельных страниц тем пока нет, но блок «Темы» на главной строится автоматически по тегам.

Для новой заметки Web-Codex должен:

- прочитать текст;
- выбрать 3–6 технически полезных тегов;
- переиспользовать существующие теги, если они подходят;
- не добавлять слишком общие теги без необходимости.

Примеры хороших тегов:

```md
tags: ["Bitrix", "Git", "CI/CD", "Astro", "AI-агенты"]
```

Карточки и темы отдельно прописывать не нужно. Они появляются из данных коллекции Astro.

## Правила работы с текстом

Если пользователь дал готовый текст заметки, Web-Codex не должен переписывать его литературно или редакторски. Допустима только корректура: явные опечатки, случайные двойные пробелы, сломанная пунктуация, технически неверные имена вроде `ci/cd` вместо `CI/CD`, если это не меняет авторский голос.

## Проверка перед PR

Перед commit нужно выполнить:

```bash
npm run build
```

Если зависимости в окружении Web-Codex еще не установлены:

```bash
npm ci
npm run build
```

Минимально проверить локальные маршруты, которые затронуты задачей. Для новой заметки это:

```text
/
/notes/
/notes/<slug>/
```

В commit не добавлять временные скриншоты из `screenshots/`, если пользователь явно не попросил сохранить их в Git.

## Если задача не про новую заметку

Для изменений дизайна, структуры страниц, CSS или логики списков Web-Codex должен:

- учитывать референсы в `design/`;
- не добавлять фейковую функциональность;
- не показывать ссылки на пустые или неработающие разделы;
- не трогать CI/CD и серверные настройки без отдельной задачи;
- после изменений запускать `npm run build`;
- открывать pull request с кратким описанием изменений и проверки.
